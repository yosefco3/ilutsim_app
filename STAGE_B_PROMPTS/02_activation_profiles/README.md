# 02 · פרופילי הפעלה (Activation Profiles)

**מטרה:** הישות הבסיסית של **חלק ב' (בונה הסידור)** — הקופסה שמחזיקה עמדות. פרופיל = **תבנית
לשימוש חוזר** ("מצב" עבודה: שגרה / חג / אירוע). נבנית לפני העמדות, כי הכל נתלה עליה.
זו גם המשימה הראשונה של חלק ב', ולכן היא **מציבה את הגבול בקוד** בין שני חלקי האפליקציה.

## ההחלטות שננעלו מול המשתמש (מקור אמת)

- **פרופיל = תבנית לשימוש חוזר, לא מופע פר-שבוע.** התרחיש המרכזי: יש פרופיל "שגרה";
  בשבוע עם חג באמצע — המשתמש **משכפל** את "שגרה", **משנה יום אחד**, ושומר כפרופיל חדש.
  → **שכפול + עריכה הם הלב של הפיצ'ר.**
- **סוג הפרופיל = טקסט חופשי** (לא enum נעול). גמישות מקסימלית.
- **העמדות עדיין לא נוצרות כאן** — הפרופיל נשאר ריק מעמדות בסוף 02 (עמדות = משימה 03).
  אבל המודל/השירות מתוכננים כך ש**בעתיד אפשר להעביר עמדות בין פרופילים** (FK ניתן לשיוך-מחדש)
  ו**שכפול יבצע deep-copy** של העמדות (נקודת הרחבה מקומית: `_copy_positions`, כרגע no-op).
- **שיוך פרופיל↔שבוע** לא נכנס כאן — מגיע עם הלוח (משימה 04).
- **הפרדה ברורה בקוד** (דרישה מפורשת): כל חלק ב' חי תחת `backend/app/schedule_builder/`
  (חבילת דומיין נפרדת) + `frontend/admin/src/pages/builder/`. תלות חד-כיוונית: ב' → א' מותר,
  א' → ב' אסור. ה-UI — הפרדה ויזואלית בקבוצת ניווט "בונה הסידור" (שיקול המבצע).

## קריטריון הצלחה (המשתמש מריץ)
נכנס למסך → רואה "שגרה" → יוצר "חג" → משכפל ל"חג שני" → משנה שם → מוחק →
מרענן את הדף → הכל נשמר ומשתקף נכון. (ובדיקת תשתית מ-00: האפליקציה לא נשברה.)

**הערה:** בסוף המשימה הפרופיל **ריק מעמדות** — זו הנחת-היסוד; העמדות נכנסות במשימה 03.

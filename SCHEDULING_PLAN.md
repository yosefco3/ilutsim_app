# תוכנית מערכת הסידור (Scheduling) — מסמך תכנון

> **סטטוס: תכנון בלבד — טרם מומש קוד.**
> מסמך זה מסכם את ההחלטות מהתייעצות לפני בניית מנוע הסידור.
> נוצר: 12 יוני 2026.

---

## הרקע — הפער שאנחנו סוגרים

המערכת היום **אוספת זמינות** (מי *יכול* לעבוד מתי — בוקר/ערב/לילה + שעות, דרך
`WeeklySubmission` / `DailyStatus`). זה צד הקלט.

**הפלט** — "סידור עבודה שבועי" כמו באקסל הקיים — הוא חיה אחרת: **שיבוץ של מאבטח
ספציפי לעמדה ספציפית, במשמרת ספציפית, עם שעות ספציפיות**. השורות בלוח הן *עמדות*
(חמ"ל, סייר 1/2, ארנונה, קומה 6, ב‑1...), לכל אחת שעות משלה ולעיתים דרישת תפקיד.

המודל הנוכחי לא יודע מה זה "עמדה" ומה זה "שיבוץ" — זו התשתית שצריך לבנות.

## ההחלטות (מאושרות בהתייעצות)

| נושא | החלטה |
|------|--------|
| **רמת אוטומציה** | חצי‑אוטומטי. מתחילים מ**לוח עריכה ידני עם אזהרות חיות** + drag & drop. אוטומציה אמיתית ("הצעה אוטומטית") — שכבה מעל, שלב אחרון בלבד. |
| **הגדרת העמדות** | קונפיג חד‑פעמי. רשימה קבועה יחסית (~30 עמדות בשבוע ממוצע, ~30 מאבטחים), עם הוספה/הסרה מדי פעם. |
| **הגדרת הכיסוי** | נגזרת מהקונפיג — כל משבצת עמדה פעילה = תא בגריד שצריך לאייש. אין הגדרת "כיסוי" נפרדת. |
| **גמישות ברמת התא** | רוב התאים סטנדרטיים. חריגים (כמו ארנונה ביום ג') = פשוט אין משבצת → אין תא. אין מנגנון "חסום" נפרד. |
| **מחזור חיים** | בונים את הסידור על שבוע בסטטוס `locked`. "פרסום" (`published`) = נעילת הסידור הסופי. |

## מבנה הקונפיג — איך המשתמש חושב על זה

הציר הראשי: **א'–ה' מול ו'–ש'**. בתוך א'–ה' יש דפוס משנה:
**א' ב' ד' ה' זהים**, ו**יום ג' חריג** (העירייה עובדת אחה"צ; בחלק מהעמדות שעות שונות
או אין בוקר).

מסך הקונפיג בנוי בבלוקים:

```
▸ בלוק "עמדות א'–ה'"
     ▸ א' ב' ד' ה'  (זהים — ממלאים פעם אחת)
     ▸ יום ג'  (חריג):  ◯ כמו השאר  ◯ שעות שונות  ◯ לא פעיל במשמרת
▸ בלוק "עמדות ו'–ש'"   (רשימה נפרדת וקצרה)
▸ בלוק "עמדות אירוע"   (אד-הוק — ישיבות מועצה, ועדות, טקסים)
```

מאחורי הקלעים הכל נשמר כ‑`PostSlot` לכל יום, אבל המשתמש ממלא פעם אחת ולא חמש.

## פרופילי הפעלה (מצבים) — שגרה / חג / מלחמה

שכבה מעל הקונפיג, למצבים שבהם "הכל משתנה":

```
ScheduleProfile
├─ "שגרה"   ← ברירת מחדל (כל הקונפיג הרגיל)
├─ "חג"     ← קבוצת עמדות/שעות משלו
├─ "מלחמה"  ← קבוצת עמדות/שעות משלו
└─ ...
```

- כל פרופיל מחזיק את העמדות והמשבצות שלו (`Post.profile_id`).
- יצירת פרופיל = **"שכפל מ‑שגרה ושנה"** — לא מתחילים מאפס.
- בבניית הסידור לשבוע בוחרים פרופיל (ברירת מחדל: שגרה). הגריד נבנה מעמדות הפרופיל.
- עמדות אירוע ועריכות אד‑הוק עובדות **מעל כל פרופיל**.
- **הנדסית:** `ScheduleProfile` + `Post.profile_id` נכנסים כבר בשלב 1 (פרופיל "שגרה"
  זרוע כברירת מחדל), כדי להימנע ממיגרציה כואבת. ניהול פרופילים (שכפול/מעבר) — שלב מאוחר.

## מודל הנתונים

```
ScheduleProfile (מצב הפעלה)
   שם, is_default

Post (עמדה)
   profile_id  →  ScheduleProfile
   שם                       "ארנונה" / "חמ"ל" / "סייר 1" / "ישיבת מועצה"
   category                 standing (קבועה)  |  event (אד-הוק)
   template                 weekday (א'–ה')  |  weekend (ו'–ש')
   requires_ahmash          כן / לא
   requires_patrol_vehicle  כן / לא  (הסמכת רכב סיור)
   קבוצת-תצוגה / סדר

PostSlot (משבצת עמדה)
   post_id  →  Post
   day_of_week  ×  shift (בוקר/ערב/לילה)  ×  start_time  ×  end_time

Assignment (שיבוץ)
   week_id   →  ScheduleWeek
   slot_id   →  PostSlot
   user_id   →  User (מאבטח)
   שעות-בפועל (override אופציונלי), סטטוס

User (קיים) — תוספת:
   patrol_vehicle_certified : bool   (מוסמך רכב סיור)
```

**דוגמה לעמדה:**
```
ארנונה
  profile: שגרה | category: standing | template: weekday
  אחמ"ש: לא | רכב סיור: לא
  משבצות:
    א' בוקר 07:30–15:00
    ב' בוקר 07:30–15:00
    ג' — רק אחה"צ (אין בוקר)
    ד' בוקר 07:30–15:00
    ה' בוקר 07:30–15:00
```

## אזהרות רכות (לא חוסמות) — מולאות בזמן עריכה

באותה פילוסופיה של האזהרות הרכות הקיימות בצד השומר:

- המאבטח **לא סימן זמינות** לאותו יום/משמרת.
- **שיבוץ כפול** — כבר משובץ בשעה חופפת בעמדה אחרת.
- **חוסר תפקיד/הסמכה** — עמדה דורשת אחמ"ש / רכב סיור והמאבטח לא עומד בזה.
- **חריגה ממקס' ימים רצוף** / מנוחה.
- **עמדה ריקה** — משבצת שעדיין לא אוישה.

כולן אינפורמטיביות; המנהל תמיד יכול לעקוף.

> כל שלב 2‑4 קבצים, נבדק ומקומיט בנפרד (לפי CLAUDE.md). שמישות אמיתית כבר משלב 4‑5.

| # | שלב | תוצר |
|---|-----|------|
| 1 | **מודל פרופיל + עמדות + משבצות + קונפיג** (backend + מיגרציה, פרופיל "שגרה" זרוע) | מגדירים את כל העמדות פעם אחת, בבלוקים א'–ה' / ו'–ש' / אירוע |

| 4 | **מילוי ידני** — קליק על תא → בחירת מאבטח (מסונן לפי זמינות) + שמירה | אפשר לשבץ |

| 7 | **עריכות אד‑הוק לשבוע** — הוספת/ביטול משבצת + עמדות אירוע לשבוע ספציפי | גמישות לחריגים (טקס, ועדה) |

## נקודות פתוחות / להחלטה בהמשך

- אלגוריתם ה"הצעה האוטומטית" (שלב 9) — greedy פשוט (זמינות + תפקיד + איזון עומס) מספיק
  להתחלה; solver אמיתי רק אם יידרש.
- UX מדויק של drag & drop מול קליק‑לבחירה — נחליט בשלב 6.
- האם פרופיל נבחר ברמת שבוע או שאפשר לערבב — כרגע: ברמת שבוע.
